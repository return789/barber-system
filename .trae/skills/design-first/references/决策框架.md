# 决策框架

## 概述

本文档详细阐述"设计先行"技能所采用的决策框架，包括结构化决策方法、权衡分析技术和决策质量评估标准。这些框架确保设计决策的科学性、可追溯性和可执行性。

---

## 1. 结构化决策流程

### 1.1 决策生命周期模型

设计先行技能采用改进版的 OODA 循环（观察-调整-决策-行动），针对软件设计场景进行了优化。

```
┌─────────────────────────────────────────────────────────────┐
│              设计决策生命周期 (D-ODA)                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────┐                                           │
│  │  D-Discover │ ← 发现/澄清阶段                            │
│  │   发现需求   │   收集上下文、约束、优先级                  │
│  └──────┬──────┘                                           │
│         │                                                   │
│         ↓                                                   │
│  ┌─────────────┐                                           │
│  │  O-Options  │ ← 提案阶段                                 │
│  │   生成选项   │   构建方案、分析权衡                        │
│  └──────┬──────┘                                           │
│         │                                                   │
│         ↓                                                   │
│  ┌─────────────┐                                           │
│  │  D-Decide   │ ← 收敛阶段                                 │
│  │   做出决策   │   评估选择、获得批准                        │
│  └──────┬──────┘                                           │
│         │                                                   │
│         ↓                                                   │
│  ┌─────────────┐                                           │
│  │  A-Act      │ ← 记录/执行阶段                            │
│  │   执行行动   │   记录决策、实施代码                        │
│  └─────────────┘                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 决策类型分类

根据决策的影响范围和可逆性，采用不同的决策流程深度：

| 决策类型 | 特征 | 示例 | 流程深度 |
|----------|------|------|----------|
| **Type 1: 不可逆决策** | 高影响、难回滚 | 架构选型、技术栈选择 | 完整四阶段 + 文档化 |
| **Type 2: 可逆决策** | 中等影响、可调整 | API 设计、数据结构 | 标准四阶段 |
| **Type 3: 低风险决策** | 低影响、易修改 | 变量命名、代码风格 | 简化流程（直接提案） |

**决策类型判断流程图**：
```
开始决策
    │
    ├── 影响范围 > 3 个模块？
    │       │
    │       ├── 是 ──→ Type 1（完整流程）
    │       │
    │       └── 否 ──┐
    │                 │
    │                 ├── 回滚成本 > 1 周？
    │                 │       │
    │                 │       ├── 是 ──→ Type 1
    │                 │       │
    │                 │       └── 否 ──┐
    │                 │                 │
    │                 │                 ├── 影响范围 > 1 个模块？
    │                 │                 │       │
    │                 │                 │       ├── 是 ──→ Type 2
    │                 │                 │       │
    │                 │                 │       └── 否 ──→ Type 3
    │                 │                 │
    │                 │                 └── 简化流程
```

## 2. 权衡分析技术

### 2.1 权衡矩阵 (Trade-off Matrix)

权衡矩阵是评估和比较不同方案的核心工具，通过量化关键维度来支持决策。

**标准权衡矩阵模板**：

| 评估维度 | 权重 | Option A | Option B | 说明 |
|----------|------|----------|----------|------|
| 实现复杂度 | 20% | ★★★☆☆ | ★★☆☆☆ | A 需要更多开发工作 |
| 维护成本 | 20% | ★★☆☆☆ | ★★★★☆ | B 长期维护更简单 |
| 性能表现 | 15% | ★★★★☆ | ★★★☆☆ | A 性能略优 |
| 可扩展性 | 15% | ★★★★★ | ★★★☆☆ | A 架构更灵活 |
| 团队熟悉度 | 15% | ★★★☆☆ | ★★★★★ | 团队更熟悉 B 技术 |
| 风险程度 | 15% | ★★★☆☆ | ★★★★☆ | B 风险更低 |
| **加权总分** | 100% | **3.45** | **3.55** | B 略优 |

**评分标准**：
```
★★★★★ = 优秀 (5分) - 完全满足需求，有显著优势
★★★★☆ = 良好 (4分) - 满足需求，有一定优势
★★★☆☆ = 一般 (3分) - 基本满足需求
★★☆☆☆ = 较差 (2分) - 部分满足需求，有明显不足
★☆☆☆☆ = 很差 (1分) - 无法满足需求
```

### 2.2 决策树分析

对于复杂的技术选型，使用决策树来可视化决策路径和结果。

```
示例：认证方案决策树

                        认证需求
                            │
            ┌───────────────┼───────────────┐
            │               │               │
        自建认证        第三方服务        混合方案
            │               │               │
    ┌───────┴───────┐       │       ┌───────┴───────┐
    │               │       │       │               │
  JWT方案      Session方案  │    主自建+备用第三方
    │               │       │       │               │
  优点:          优点:     优点:   优点:
  - 无状态       - 易控制   - 快速集成 - 灵活性高
  - 易扩展       - 即时失效 - 维护少  - 有备份
    │               │       │       │
  缺点:          缺点:     缺点:   缺点:
  - 难撤销       - 需存储   - 依赖外部 - 复杂度高
  - Token大      - 扩展难  - 成本高  - 开发量大
```

## 3. 方案生成与收敛技术

### 3.1 方案生成原则

**为什么是 2 个方案？**

设计先行技能强制要求提出"正好 2 个方案"，这一限制基于以下理论和实践考量：

| 理论基础 | 解释 |
|----------|------|
| **认知负荷理论** | 工作记忆限制约 4-7 个信息块，比较 2 个方案各需占用 1-2 个块，接近最优 |
| **决策瘫痪研究** | 选项过多会导致决策延迟和质量下降（Hick's Law） |
| **对比效应** | 两个方案的对比能更清晰地展现各自优劣 |
| **实用主义** | 强制聚焦于最可行的方案，避免"为了凑数"的低质量选项 |

**方案差异化要求**：
```
Option A 和 Option B 必须有本质差异：

✅ 好的方案对：
- 自建 vs 第三方服务
- 同步处理 vs 异步队列
- SQL 数据库 vs NoSQL 数据库

❌ 不好的方案对：
- 使用 Redis vs 使用 Memcached（差异太小）
- 方案 A vs 方案 A 的微调版本（不是真正的选择）
```

### 3.2 SCAMPER 创新法

在生成方案时，可以使用 SCAMPER 技术来扩展思路：

```
S - Substitute (替代)
    └── 能否用其他技术/方法替代？

C - Combine (组合)
    └── 能否组合多种方案的优势？

A - Adapt (调整)
    └── 能否调整现有方案适应新需求？

M - Modify (修改)
    └── 能否修改参数/配置优化方案？

P - Put to other uses (其他用途)
    └── 现有方案能否用于其他场景？

E - Eliminate (消除)
    └── 能否消除不必要的复杂性？

R - Reverse (反转)
    └── 能否反转某些假设或顺序？
```

### 3.3 收敛决策规则

**迭代限制机制**：

设计先行技能限制最多 2 轮迭代，避免无限循环。这一规则基于以下考量：

```
┌─────────────────────────────────────────────────────────────┐
│  迭代收敛规则                                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Round 1: 初始提案 → 被拒绝                                  │
│      │                                                      │
│      ├── 拒绝原因明确？                                      │
│      │       │                                              │
│      │       ├── 是 ──→ 针对性修改 → Round 2                 │
│      │       │                                              │
│      │       └── 否 ──→ 追问具体原因                         │
│      │                                                      │
│  Round 2: 修改提案 → 被拒绝                                  │
│      │                                                      │
│      └── 触发升级机制：                                      │
│          "我们已探索 2 轮方案，建议：                         │
│           a) 您提供具体方向                                   │
│           b) 安排会议讨论                                    │
│           c) 由我基于最佳判断执行"                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 4. 决策质量评估

### 4.1 决策质量框架 (Decision Quality Framework)

每个设计决策应满足以下六个维度：

```
┌─────────────────────────────────────────────────────────────┐
│                    决策质量六维度                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 适当的框架 (Appropriate Frame)                          │
│     └── 是否正确定义了问题？                                 │
│     └── 是否考虑了正确的范围？                               │
│                                                             │
│  2. 创造性的选项 (Creative Alternatives)                    │
│     └── 是否有足够多的可行方案？                             │
│     └── 方案之间是否有本质差异？                             │
│                                                             │
│  3. 相关可靠的信息 (Relevant & Reliable Information)        │
│     └── 是否基于充分的信息？                                 │
│     └── 信息来源是否可靠？                                   │
│                                                             │
│  4. 清晰的价值权衡 (Clear Values and Trade-offs)            │
│     └── 是否明确了各方案的优劣？                             │
│     └── 是否量化了关键权衡？                                 │
│                                                             │
│  5. 充分的论证 (Sound Reasoning)                            │
│     └── 推荐理由是否充分？                                   │
│     └── 是否考虑了边界情况？                                 │
│                                                             │
│  6. 承诺执行 (Commitment to Action)                         │
│     └── 决策是否可执行？                                     │
│     └── 是否有明确的下一步？                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 决策记录模板

**标准决策记录格式**：

```markdown
## 设计决策记录 (ADR)

### 决策标题
[简短描述决策内容]

### 状态
[提议中 / 已批准 / 已实施 / 已废弃]

### 背景
[描述导致此决策的背景和问题]

### 决策
[描述做出的决策和理由]

### 方案对比
| 维度 | Option A | Option B |
|------|----------|----------|
| ... | ... | ... |

### 后果
**正面影响**:
- ...

**负面影响**:
- ...

**风险**:
- ...

### 相关决策
- [关联的其他决策记录]
```

### 4.3 决策复盘机制

对于重要决策，建议在实施后进行复盘：

```
复盘时间点：
- Type 1 决策：实施后 1 个月
- Type 2 决策：实施后 2 周
- Type 3 决策：无需复盘

复盘问题清单：
1. 决策是否达到了预期目标？
2. 实际后果与预期是否一致？
3. 有哪些未预见的问题？
4. 如果重来，会做出不同决策吗？
5. 有哪些经验可以沉淀？
```

## 5. 群体决策支持

### 5.1 决策角色定义

在涉及多方参与的决策中，明确角色和职责：

| 角色 | 职责 | 权限 |
|------|------|------|
| **决策者** | 做出最终决定 | 拥有最终决策权 |
| **建议者** | 提供专业意见 | 提供方案和分析 |
| **执行者** | 实施决策 | 提出实施可行性反馈 |
| **受影响者** | 被告知决策 | 提供使用场景反馈 |

### 5.2 共识达成技术

当需要团队共识时，采用以下技术：

**拇指投票法 (Fist of Five)**：
```
5 指：完全同意，愿意积极支持
4 指：同意，愿意支持
3 指：中立，可以接受
2 指：有保留，需要讨论
1 指：不同意，需要重大修改
拳头：完全反对，阻止决策

决策规则：
- 所有人 3 指以上：决策通过
- 有人 2 指或以下：需要讨论并解决顾虑
```

## 6. 决策陷阱与规避

### 6.1 常见决策陷阱

| 陷阱 | 描述 | 规避策略 |
|------|------|----------|
| **锚定效应** | 过度依赖第一个获得的信息 | 多方案对比，延迟判断 |
| **确认偏误** | 只寻找支持已有观点的证据 | 主动寻找反对证据 |
| **沉没成本谬误** | 因已投入而继续错误决策 | 评估未来成本，忽略过去投入 |
| **分析瘫痪** | 过度分析导致无法决策 | 设定决策时限，接受"足够好" |
| **群体思维** | 为和谐而压制异议 | 指定"魔鬼代言人"角色 |
| **近因效应** | 过度重视最近的信息 | 系统性回顾所有相关信息 |

### 6.2 决策检查清单

在做出重要决策前，使用此检查清单：

```
□ 是否正确定义了问题？
□ 是否考虑了至少 2 个本质不同的方案？
□ 是否收集了足够的信息？
□ 是否识别并评估了风险？
□ 是否考虑了长期影响？
□ 是否有明确的执行计划？
□ 是否记录了决策理由？
□ 是否设定了复盘时间点？
```
