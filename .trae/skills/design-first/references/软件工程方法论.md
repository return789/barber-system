# 软件工程方法论

## 概述

本文档汇总了"设计先行"技能所依据的核心软件工程方法论和原则，为技能的实施提供理论支撑和实践指导。

---

## 1. YAGNI 原则 (You Ain't Gonna Need It)

### 定义

YAGNI 是极限编程(XP)中的核心原则之一，由 Ron Jeffries 提出。其核心思想是：**只实现当前真正需要的功能，不要为未来可能需要的功能预先编码**。

### 理论基础

```
┌─────────────────────────────────────────────────────────────┐
│  YAGNI 的经济学逻辑                                          │
├─────────────────────────────────────────────────────────────┤
│  成本 = 实现成本 + 维护成本 + 机会成本                        │
│                                                             │
│  预先实现的功能：                                            │
│  - 实现成本：立即支付                                        │
│  - 维护成本：持续支付（即使从未使用）                         │
│  - 机会成本：占用了实现真正需要功能的时间                     │
│                                                             │
│  延迟实现的功能：                                            │
│  - 实现成本：仅在需要时支付                                   │
│  - 维护成本：零（直到实现）                                   │
│  - 机会成本：可能永远不需要实现                               │
└─────────────────────────────────────────────────────────────┘
```

### 在设计先行中的应用

| 场景 | YAGNI 指导 | 实践建议 |
|------|-----------|----------|
| 架构设计 | 只为当前需求设计架构 | 避免过度抽象和过度工程化 |
| API 设计 | 只暴露当前需要的接口 | 预留扩展点但不预先实现 |
| 数据模型 | 只存储当前需要的字段 | 使用灵活的数据结构应对变化 |
| 代码结构 | 只创建当前需要的模块 | 延迟模块化决策 |

### 参考文献

- **原始来源**: Ron Jeffries, Extreme Programming Installed (2001)
- **扩展阅读**: Martin Fowler, "Is High Quality Software Worth the Cost?" (2019)
- **相关原则**: KISS (Keep It Simple, Stupid), DRY (Don't Repeat Yourself)

## 2. 最小可行产品 (MVP)

### 定义

MVP 概念由 Frank Robinson 首创，后经 Eric Ries 在《精益创业》中推广。其定义为：**用最少的开发工作，获取最大量的有效学习**。

### MVP 的三种类型

```
类型 A: 验证问题是否存在
├── 目标：确认用户真的有这个问题
├── 方法：访谈、问卷、观察
└── 产出：问题验证报告

类型 B: 验证解决方案是否可行
├── 目标：确认方案能解决问题
├── 方法：原型、演示、Wizard of Oz
└── 产出：方案可行性评估

类型 C: 验证产品是否被接受
├── 目标：确认用户愿意使用/付费
├── 方法：功能受限版本、Beta 测试
└── 产出：市场验证数据
```

### 在设计先行中的应用

**提案阶段的 MVP 思维**：
- 优先推荐能快速验证假设的方案
- 评估方案时考虑"最小可行"版本
- 避免一次性设计"完美"解决方案

**决策记录中的 MVP 标注**：
```
// Design Decision: 2024-03-20
// Chosen: Option A - 简单实现，快速验证
// MVP 范围: 仅支持单一认证方式，后续迭代添加其他方式
// 验证目标: 确认用户愿意使用新的认证流程
```

### 参考文献

- **核心著作**: Eric Ries, The Lean Startup (2011)
- **原始论文**: Frank Robinson, "The MVP Strategy" (2001)
- **扩展应用**: Steve Blank, The Four Steps to the Epiphany (2005)

## 3. 渐进式架构 (Evolutionary Architecture)

### 定义

渐进式架构是一种支持在技术栈、功能需求和业务环境变化时，能够持续、增量地适应和演进的架构方法。由 Neal Ford、Rebecca Parsons 和 Patrick Kua 系统化提出。

### 核心概念

**架构量子 (Architecture Quantum)**：
> 能够独立部署的最小架构单元，具有高度内聚和松耦合特性。

**适应度函数 (Fitness Functions)**：
> 用于评估架构是否满足业务需求和技术目标的客观度量标准。

```
┌─────────────────────────────────────────────────────────────┐
│  渐进式架构决策框架                                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  业务需求 ──→ 适应度函数 ──→ 架构决策                        │
│      │              │              │                        │
│      │              ↓              ↓                        │
│      │         评估指标      架构量子边界                    │
│      │              │              │                        │
│      └──────────────┴──────────────┘                        │
│                    │                                        │
│                    ↓                                        │
│              持续验证与演进                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 在设计先行中的应用

**提案阶段的适应度函数**：
```markdown
## Option A: 微服务架构

**适应度函数评估**:
- 部署独立性: ★★★★★ (高)
- 故障隔离性: ★★★★☆ (中高)
- 开发复杂度: ★★☆☆☆ (低)
- 运维成本: ★★☆☆☆ (低)

**权衡**: 提高部署灵活性，增加运维复杂度
```

### 参考文献

- **核心著作**: Neal Ford et al., Building Evolutionary Architectures (2017)
- **相关概念**: Martin Fowler, "Patterns of Enterprise Application Architecture" (2002)
- **实践指南**: ThoughtWorks Technology Radar

## 4. 技术债务管理

### 定义

技术债务概念由 Ward Cunningham 于 1992 年提出，比喻为：**为了短期速度而做出的技术妥协，最终需要支付"利息"（额外维护成本）**。

### 债务分类

| 类型 | 描述 | 管理策略 |
|------|------|----------|
| **故意债务** | 有意识地为了速度而妥协 | 记录债务，计划偿还时间 |
| **无意债务** | 由于知识不足或时间压力产生 | 代码审查时识别，逐步重构 |
| **腐化债务** | 环境变化导致的设计过时 | 定期评估架构适应性 |

### 债务量化模型

```
技术债务成本 = 本金 + (利息 × 时间)

其中：
- 本金 = 消除债务所需的工作量
- 利息 = 债务导致的额外维护成本/单位时间
- 时间 = 债务存在的时间

决策规则：
- 当 利息 × 预期存在时间 > 本金 时，应该偿还债务
- 否则，可以暂时保留债务
```

### 在设计先行中的应用

**提案阶段的技术债务评估**：
```markdown
## Option A: 快速实现方案

**技术债务分析**:
- 债务类型: 故意债务
- 债务本金: 约 20 人时（后续重构成本）
- 预期利息: 2 人时/月（额外维护成本）
- 偿还计划: Q2 重构迭代

**决策**: 接受此债务以支持 MVP 发布
```

### 参考文献

- **原始概念**: Ward Cunningham, "The WyCash Portfolio Management System" (1992)
- **扩展研究**: Steve McConnell, "Technical Debt" (2007)
- **管理框架**: Martin Fowler, "Technical Debt Quadrant" (2009)

## 5. 设计思维 (Design Thinking)

### 定义

设计思维是一种以人为本的创新方法论，由 IDEO 和斯坦福大学 d.school 推广。其核心是：**通过同理心理解用户需求，通过迭代实验验证解决方案**。

### 五阶段模型

```
┌─────────────────────────────────────────────────────────────┐
│                    设计思维五阶段                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 同理心 (Empathize)                                      │
│     └── 深入理解用户需求和痛点                               │
│              │                                              │
│              ↓                                              │
│  2. 定义 (Define)                                           │
│     └── 明确要解决的核心问题                                 │
│              │                                              │
│              ↓                                              │
│  3. 构思 (Ideate)                                           │
│     └── 生成多种可能的解决方案                               │
│              │                                              │
│              ↓                                              │
│  4. 原型 (Prototype)                                        │
│     └── 快速构建可测试的原型                                 │
│              │                                              │
│              ↓                                              │
│  5. 测试 (Test)                                             │
│     └── 验证方案并迭代改进                                   │
│              │                                              │
│              └──→ 循环迭代                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 在设计先行中的应用

**发现阶段的设计思维映射**：
- 同理心 → 理解用户真实需求和使用场景
- 定义 → 明确问题的边界和成功标准
- 构思 → 提出多个可行方案（提案阶段）

**提案阶段的收敛思维**：
- 从发散的多个方案中收敛到 2 个最佳选项
- 每个方案都基于用户价值进行评估

### 参考文献

- **核心著作**: Tim Brown, Change by Design (2009)
- **方法论**: IDEO, "Human-Centered Design Toolkit"
- **教育实践**: Stanford d.school, "Design Thinking Bootleg"

## 6. 认知负荷理论

### 定义

认知负荷理论由 John Sweller 于 1988 年提出，研究工作记忆在学习和问题解决中的限制。核心发现：**工作记忆容量有限（约 4-7 个信息块），过载会导致理解和决策能力下降**。

### 三种认知负荷类型

| 类型 | 来源 | 影响 | 管理策略 |
|------|------|------|----------|
| **内在负荷** | 任务本身的复杂性 | 不可避免 | 分解任务，逐步处理 |
| **外在负荷** | 信息呈现方式不当 | 应该减少 | 优化信息组织，使用可视化 |
| **相关负荷** | 学习和理解过程 | 应该增加 | 提供上下文，建立关联 |

### 在设计先行中的应用

**批量提问的认知科学依据**：
```
传统方式（多轮零散提问）:
问题1 → 等待 → 问题2 → 等待 → 问题3 → 等待
认知负荷：多次上下文切换，每次都需要重建心智模型

设计先行方式（批量提问）:
问题1 + 问题2 + 问题3 → 统一等待
认知负荷：单次上下文建立，降低外在认知负荷
```

**方案数量的限制**：
- 为什么是 2 个方案而不是更多？
- 工作记忆限制：比较 2 个方案需要保持 2 个心智模型
- 超过 2 个会导致认知过载，决策质量下降

### 参考文献

- **原始论文**: John Sweller, "Cognitive Load During Problem Solving" (1988)
- **扩展应用**: Daniel Kahneman, Thinking, Fast and Slow (2011)
- **软件工程应用**: Felienne Hermans, "The Programmer's Brain" (2021)
