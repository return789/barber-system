# 敏捷开发实践

## 概述

本文档阐述"设计先行"技能与敏捷开发方法的融合实践，包括迭代开发、持续交付、反馈循环等核心概念在技能实施中的应用。

---

## 1. 敏捷宣言与设计先行

### 1.1 敏捷宣言核心价值映射

设计先行技能的设计理念与敏捷宣言的四条核心价值高度契合：

```
┌─────────────────────────────────────────────────────────────┐
│           敏捷宣言价值 → 设计先行实践映射                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  个体和互动 高于 流程和工具                                   │
│  │                                                          │
│  └──→ 批量提问而非零散追问                                   │
│       重视用户反馈而非机械流程                                │
│       思考过程 > 文档文件                                    │
│                                                             │
│  工作的软件 高于 详尽的文档                                   │
│  │                                                          │
│  └──→ 速度优于形式                                           │
│       行内注释记录而非独立文档                                │
│       偏向行动而非过度规划                                    │
│                                                             │
│  客户合作 高于 合同谈判                                       │
│  │                                                          │
│  └──→ 用户批准后才执行                                       │
│       方案选择权在用户                                        │
│       最多 2 轮迭代避免僵局                                   │
│                                                             │
│  响应变化 高于 遵循计划                                       │
│  │                                                          │
│  └──→ YAGNI 原则                                             │
│       渐进式架构                                             │
│       动态调整思考深度                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 敏捷原则在技能中的体现

| 敏捷原则 | 设计先行实践 |
|----------|--------------|
| 最高优先级是通过尽早和持续交付有价值的软件满足客户 | 快速进入编码阶段（批准后立即执行） |
| 欣然面对需求变化，即使在开发后期 | 方案设计考虑扩展性，但不过度设计 |
| 经常交付可工作的软件，周期越短越好 | 小任务直接提案，大任务分阶段设计 |
| 业务人员和开发人员必须每天一起工作 | AI 与用户紧密协作，批量澄清需求 |
| 围绕有激情的个人构建项目 | AI 提供建议，用户做最终决策 |
| 面对面交谈是最有效的沟通方式 | 结构化问答，避免来回拉扯 |
| 可工作的软件是进度的首要度量标准 | 设计完成后立即产出代码 |
| 可持续的开发速度 | 避免过度设计导致的返工 |
| 持续关注技术卓越和良好设计 | 权衡分析确保设计质量 |
| 简洁——最大化不做事的艺术 | YAGNI 原则，只设计当前需要的 |
| 最好的架构、需求和设计出自自组织团队 | 用户主导决策方向 |
| 团队定期反思如何提高效率 | 决策复盘机制 |

## 2. 迭代开发与增量设计

### 2.1 迭代设计模型

设计先行技能采用增量设计方法，将大型设计任务分解为可管理的迭代：

```
┌─────────────────────────────────────────────────────────────┐
│                    增量设计迭代模型                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  迭代 0: 架构骨架                                            │
│  ├── 目标：确定核心架构方向                                   │
│  ├── 产出：架构决策记录、技术选型                             │
│  └── 时间：完整四阶段流程                                     │
│                                                             │
│  迭代 1: 核心功能                                            │
│  ├── 目标：实现 MVP 核心功能                                  │
│  ├── 产出：可工作的核心模块                                   │
│  └── 时间：简化流程（基于迭代 0 的架构决策）                   │
│                                                             │
│  迭代 2: 功能扩展                                            │
│  ├── 目标：添加次要功能                                       │
│  ├── 产出：扩展的功能集                                       │
│  └── 时间：按需决策流程                                       │
│                                                             │
│  迭代 N: 优化完善                                            │
│  ├── 目标：性能优化、体验改善                                 │
│  ├── 产出：优化后的系统                                       │
│  └── 时间：简化流程                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 设计决策的迭代粒度

根据任务规模动态调整设计决策的粒度：

```
任务规模评估矩阵：

┌──────────────┬──────────────┬──────────────┬──────────────┐
│   小型任务    │   中型任务    │   大型任务    │   超大型任务  │
├──────────────┼──────────────┼──────────────┼──────────────┤
│ 1-2 文件     │ 3-10 文件    │ 10-50 文件   │ > 50 文件    │
│ < 100 行代码 │ 100-500 行   │ 500-2000 行  │ > 2000 行    │
│ 单一模块     │ 2-3 模块     │ 多模块       │ 跨系统       │
├──────────────┼──────────────┼──────────────┼──────────────┤
│ 直接提案     │ 标准流程     │ 完整流程     │ 分阶段流程   │
│ 跳过发现阶段 │ 2-3 问题     │ 3+ 问题      │ 多轮发现     │
│ 快速批准     │ 单次批准     │ 详细批准     │ 分层批准     │
└──────────────┴──────────────┴──────────────┴──────────────┘
```

### 2.3 滚动式规划

设计先行采用滚动式规划，只详细设计当前迭代，后续迭代保持高层规划：

```
当前迭代（详细设计）
├── 完整四阶段流程
├── 详细权衡分析
├── 明确的实现计划
└── 具体的技术选型

下一迭代（概要规划）
├── 功能范围定义
├── 依赖关系识别
├── 风险评估
└── 技术方向（待细化）

未来迭代（愿景规划）
├── 产品愿景
├── 技术路线图
├── 关键里程碑
└── 主要风险
```

## 3. 持续交付与快速反馈

### 3.1 设计反馈循环

设计先行技能建立了快速反馈循环，确保设计方向正确：

```
┌─────────────────────────────────────────────────────────────┐
│                    设计反馈循环                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│     ┌─────────────┐                                         │
│     │  用户需求    │                                         │
│     └──────┬──────┘                                         │
│            │                                                │
│            ↓                                                │
│     ┌─────────────┐      ┌─────────────┐                   │
│     │  批量提问    │ ───→ │  用户回答    │                   │
│     └──────┬──────┘      └──────┬──────┘                   │
│            │                    │                          │
│            │    ← 快速反馈循环 → │                          │
│            │                    │                          │
│            ↓                    │                          │
│     ┌─────────────┐             │                          │
│     │  方案提案    │             │                          │
│     └──────┬──────┘             │                          │
│            │                    │                          │
│            ↓                    │                          │
│     ┌─────────────┐      ┌─────────────┐                   │
│     │  用户审批    │ ───→ │  批准/拒绝   │                   │
│     └──────┬──────┘      └──────┬──────┘                   │
│            │                    │                          │
│            │    ← 决策反馈循环 → │                          │
│            │                    │                          │
│            ↓                    │                          │
│     ┌─────────────┐             │                          │
│     │  执行编码    │ ←──────────┘                          │
│     └──────┬──────┘                                         │
│            │                                                │
│            ↓                                                │
│     ┌─────────────┐                                         │
│     │  交付成果    │                                         │
│     └─────────────┘                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 反馈时效性原则

设计决策的反馈应遵循时效性原则：

| 反馈类型 | 时效要求 | 说明 |
|----------|----------|------|
| 需求澄清 | 单次会话内 | 批量提问，一次性获取答案 |
| 方案审批 | 1 个工作日内 | 避免设计上下文丢失 |
| 实施反馈 | 即时 | 编码过程中发现设计问题立即反馈 |
| 效果评估 | 1-2 周内 | 验证设计决策的实际效果 |

### 3.3 快速失败机制

设计先行鼓励"快速失败"而非"延迟失败"：

```
传统模式：
需求 → 设计 → 开发 → 测试 → 发现问题 → 返工
（问题在后期发现，返工成本高）

设计先行模式：
需求 → 快速设计 → 快速审批 → 快速开发 → 快速验证
（问题在设计阶段暴露，修正成本低）

快速失败原则：
1. 尽早暴露设计风险
2. 小步快跑，快速验证
3. 失败成本低，学习价值高
4. 调整方向快，机会成本低
```

## 4. 用户故事与设计决策

### 4.1 用户故事映射到设计流程

用户故事是敏捷开发的核心实践，设计先行将其融入设计流程：

```
用户故事格式：
作为 [角色]
我想要 [功能]
以便于 [价值]

↓ 映射到设计先行 ↓

Discovery 阶段：
- 角色 → 明确用户上下文
- 功能 → 理解需求范围
- 价值 → 确定成功标准

Proposal 阶段：
- 基于用户故事生成方案
- 评估方案对用户价值的贡献
- 权衡实现成本与用户价值
```

### 4.2 验收标准驱动设计

用户故事的验收标准(AC)直接指导方案设计：

```markdown
用户故事：用户登录功能

验收标准：
1. 用户可以使用邮箱和密码登录
2. 登录失败时显示明确的错误提示
3. 连续失败 5 次后锁定账户 15 分钟
4. 支持记住登录状态 7 天

↓ 设计决策 ↓

方案 A：Session-based 认证
- 优点：易于实现锁定逻辑
- 缺点：需要服务器存储

方案 B：JWT + Redis 认证
- 优点：无状态，易扩展
- 缺点：锁定逻辑稍复杂

推荐：方案 B
理由：验收标准 4 需要长期状态保持，JWT 更适合
```

### 4.3 INVEST 原则与设计粒度

用户故事的 INVEST 原则同样适用于设计决策的粒度控制：

| INVEST 原则 | 设计应用 |
|-------------|----------|
| **I**ndependent（独立的） | 设计决策应相互独立，避免级联依赖 |
| **N**egotiable（可协商的） | 方案可根据反馈调整 |
| **V**aluable（有价值的） | 设计决策应直接贡献用户价值 |
| **E**stimable（可估算的） | 设计复杂度应可评估 |
| **S**mall（小的） | 单个设计决策范围不宜过大 |
| **T**estable（可测试的） | 设计决策应有明确的验证标准 |

```
│                    设计决策看板                              │

│    待发现     │    待提案    │    待批准    │    已批准     │
├──────────────┼──────────────┼──────────────┼───────────────┤
│ ┌──────────┐ │ ┌──────────┐ │ ┌──────────┐ │ ┌──────────┐  │

│ │ 设计     │ │ │ 设计     │ │ │ 选型     │ │ │ 决策     │  │
│ └──────────┘ │ └──────────┘ │ └──────────┘ │ └──────────┘  │
│              │              │              │               │
│ ┌──────────┐ │ ┌──────────┐ │              │ ┌──────────┐  │
│ │ 缓存策略  │ │ │ 日志系统  │ │              │ │ 前端架构 │  │
│ │ 设计     │ │ │ 设计     │ │              │ │ 决策     │  │
│ └──────────┘ │ └──────────┘ │              │ └──────────┘  │
│              │              │              │               │
└──────────────┴──────────────┴──────────────┴───────────────┘
        ↑              ↑              ↑              ↑
     Discovery      Proposal     Convergence     Capture
      阶段           阶段          阶段           阶段
```

### 5.2 在制品限制 (WIP Limit)

设计决策流程应设置合理的在制品限制：

```
WIP 限制建议：

待发现：无限制
└── 允许积累待处理的设计需求

待提案：最多 3 个
└── 避免同时准备过多方案

待批准：最多 2 个
└── 确保及时获取用户反馈

已批准：无限制
└── 批准后立即进入开发
```

### 5.3 流动效率指标

监控设计决策的流动效率：

| 指标 | 定义 | 目标值 | 说明 |
|------|------|--------|------|
| **周期时间** | 从进入待发现到已批准的时间 | < 2 天 | 设计决策的整体效率 |
| **停留时间** | 在每个阶段的平均停留时间 | 各阶段 < 0.5 天 | 识别瓶颈阶段 |
| **吞吐量** | 每周完成的设计决策数量 | 根据团队规模 | 设计产能 |
| **返工率** | 被拒绝后需要重新提案的比例 | < 20% | 方案质量指标 |

## 6. 敏捷估算与规划

### 6.1 设计复杂度估算

使用相对估算方法评估设计决策的复杂度：

```
设计复杂度点数（类似故事点）：

1 点 - 简单设计
├── 单一选择，无明显权衡
├── 影响范围 < 2 个文件
└── 示例：变量命名、简单函数签名

2 点 - 中等设计
├── 2-3 个可行方案
├── 影响范围 2-5 个文件
└── 示例：数据结构选择、API 设计

3 点 - 复杂设计
├── 多个方案，复杂权衡
├── 影响范围 5-10 个文件
└── 示例：模块划分、数据流设计

5 点 - 重要设计
├── 架构级别决策
├── 影响范围 > 10 个文件
└── 示例：技术栈选型、架构模式

8 点 - 关键设计
├── 系统级别决策
├── 影响整个系统
└── 示例：微服务拆分、数据库选型
```

### 6.2 设计规划扑克

团队协作设计时，可使用规划扑克方法：

```
设计规划扑克流程：

1. 主持人介绍设计需求
2. 团队成员提问澄清（批量）
3. 每人独立选择复杂度卡片
4. 同时亮牌
5. 最高和最低者解释理由
6. 讨论达成共识
7. 记录复杂度点数

卡片序列：1, 2, 3, 5, 8, 13, 21
（斐波那契数列，体现不确定性递增）
```

### 6.3 设计速度 (Design Velocity)

跟踪团队的设计产能：

```
设计速度计算：

设计速度 = Σ(已完成设计的复杂度点数) / 迭代数

示例：
迭代 1：完成 1+2+3 = 6 点
迭代 2：完成 2+2+5 = 9 点
迭代 3：完成 3+5+3 = 11 点

平均设计速度 = (6+9+11) / 3 = 8.67 点/迭代

用途：
- 预测未来迭代可完成的设计工作量
- 识别设计产能变化趋势
- 支撑迭代规划
```

## 7. 持续改进实践

### 7.1 设计回顾会议

定期进行设计回顾，持续改进设计流程：

```
设计回顾议程（60 分钟）：

1. 数据回顾（10 分钟）
   - 周期时间趋势
   - 返工率统计
   - 设计速度变化

2. 做得好的（15 分钟）
   - 收集正面反馈
   - 识别成功模式

3. 需要改进的（15 分钟）
   - 收集改进建议
   - 识别痛点问题

4. 行动计划（15 分钟）
   - 选择 1-2 个改进项
   - 制定具体行动

5. 总结（5 分钟）
   - 确认行动负责人
   - 设定检查时间
```

### 7.2 设计实践社区

建立设计实践社区，分享设计经验：

```
实践社区活动：

定期分享会（每两周）
├── 设计案例分享
├── 权衡分析讨论
└── 新技术评估

设计评审（按需）
├── 重要设计决策评审
├── 架构方案讨论
└── 技术债务评估

知识沉淀
├── 设计模式库
├── 决策记录模板
└── 最佳实践文档
```

### 7.3 实验性改进

鼓励团队进行实验性改进：

```
改进实验框架：

假设：[改进措施] 将 [提升什么指标]

实验设计：
- 对照组：当前做法
- 实验组：新做法
- 测量指标：[具体指标]
- 实验周期：[时间长度]

结果分析：
- 数据对比
- 定性反馈
- 统计显著性

决策：
- 采纳（效果显著）
- 调整（部分有效）
- 放弃（无效或负面）
```

## 8. 与 DevOps 的集成

### 8.1 设计决策即代码

将设计决策融入 DevOps 流程：

```
设计决策流水线：

代码提交 → 设计决策检查 → CI/CD 流水线
              │
              ├── 检查是否有对应的决策记录
              ├── 检查决策是否已批准
              └── 检查实施是否符合决策

未通过检查 → 阻止合并，提示补充设计决策
```

### 8.2 基础设施即代码的设计

基础设施设计同样遵循设计先行原则：

```markdown
## 基础设施设计决策

### 背景
需要为新服务配置云基础设施

### 方案对比

| 维度 | Option A: 手动配置 | Option B: IaC (Terraform) |
|------|-------------------|---------------------------|
| 可重复性 | 低 | 高 |
| 版本控制 | 无 | 有 |
| 学习成本 | 低 | 中 |
| 维护成本 | 高 | 低 |

### 决策
选择 Option B: Terraform

### 理由
- 符合 GitOps 实践
- 支持环境一致性
- 便于审计和回滚
```

### 8.3 监控与反馈

建立设计决策的监控反馈机制：

```
设计决策监控指标：

实施指标：
├── 设计决策实施率
├── 实施偏差率
└── 实施时间偏差

效果指标：
├── 设计目标达成率
├── 相关缺陷数量
└── 性能指标变化

反馈闭环：
监控数据 → 分析偏差 → 识别原因 → 更新设计实践
```
